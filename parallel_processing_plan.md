# AI 음성 차팅 병렬 처리 파이프라인 구축 기획안
이 파일은 여러 대의 음성 분석 요청 및 대용량 청크들을 동시에 안정적으로 병렬 처리하기 위한 프론트엔드와 백엔드의 확장 설계 계획을 담고 있습니다.

---

## 1️⃣ 기획 배경 및 목표
- 다수의 강사가 동시에 다량의 차팅 분석을 요청하거나, 10시간 이상의 배치 녹음이 쪼개져 다량의 청크가 들어올 때 병목 현상 및 외부 API(Soniox, OpenAI) 한도 초과 오류를 예방합니다.
- 동시 처리 가능한 오디오 분석 수를 극대화하면서도 서버 리소스와 비용을 안정적으로 제어하는 것을 목표로 합니다.

## 2️⃣ 아키텍처 및 상세 반영 계획

### 2-1. 프론트엔드: 동시 업로드 제한 큐 (Upload Queue) 도입
- **문제점**: 10시간의 녹음을 30분 단위로 청킹하면 총 20개의 청크가 생성됩니다. 브라우저에서 20개의 대용량 파일 업로드를 한꺼번에 쏘면 네트워크 소켓 부족 및 대역폭 점유로 일부 업로드가 끊깁니다.
- **반영 계획**:
  - `p-limit` 라이브러리를 적용하여 **최대 동시 업로드 개수를 3개**로 제어합니다.
  - 1~3번 청크가 업로드 완료되면 4번 청크가 대기 큐에서 나와 순차적으로 병렬 처리되도록 프론트엔드 오디오 업로드 파이프라인을 제어합니다.

### 2-2. 백엔드: API 지수 백오프 (Exponential Backoff) 및 재시도 메커니즘
- **문제점**: 다수의 병렬 분석이 요청되면 Soniox와 OpenAI의 동시 요청수(Concurrency Limit) 초과 에러(HTTP 429 Too Many Requests)가 발생하여 분석이 실패로 바뀔 수 있습니다.
- **반영 계획**:
  - `process-chunk` 및 `poll-jobs` Deno Edge Functions 내부의 API 호출부에 자동 재시도 헬퍼 함수를 적용합니다.
  - API 실패 시 `[1초, 2초, 4초, 8초]`와 같이 대기 시간을 지수적으로 늘려가며 최대 3~5회 재시도하도록 보강하여 장애 복구력을 향상시킵니다.

### 2-3. 데이터베이스: 크레딧 차감 정합성 보장 (낙관적 락 고도화)
- **문제점**: 동일 사용자가 동시에 여러 브라우저 탭에서 병렬로 분석을 요청할 때 크레딧이 갱신되는 순간 경쟁 상태(Race Condition)가 발생해 쿼터가 마이너스로 내려가거나 초과 사용될 수 있습니다.
- **반영 계획**:
  - Supabase Database 레벨에서 크레딧 차감을 위한 트랜잭션 함수(RPC)를 정의하여, 단일 호출 내에서 안전하게 원자적(Atomic)으로 처리되도록 데이터 무결성을 보장합니다.

## 3️⃣ 기대 효과
- **안정성**: 네트워크 끊김 및 API 레이트 리밋에 대응하여 전체 분석 실패율 0.1% 이하 유지.
- **사용자 경험**: 다량의 청크 업로드 중에도 브라우저가 버벅거리지 않고 이쁘게 차례대로 처리됨.
- **확장성**: 서버 사양 업그레이드 없이 인스턴스가 실시간으로 확장되어 무제한 병렬 수용 가능.
